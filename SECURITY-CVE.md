# IPtctl CVE（公共漏洞和暴露）记录

本文档记录 IPtctl 项目相关的安全漏洞和 CVE 信息。

## CVE 管理流程

### 1. CVE 编号申请

当发现安全漏洞时，按以下流程申请 CVE 编号：

1. **漏洞确认**：验证漏洞真实存在
2. **影响评估**：评估漏洞的严重程度和影响范围
3. **修复准备**：准备修复补丁
4. **CVE 申请**：通过 MITRE 或 GitHub 申请 CVE 编号
5. **安全公告**：发布安全公告
6. **更新记录**：更新本文档

### 2. CVE 信息格式

每个 CVE 记录包含以下信息：

- CVE 编号
- 漏洞描述
- 影响版本
- 修复版本
- CVSS 评分
- 公开日期
- 参考链接

### 3. 严重程度分级

使用 CVSS v3.1 评分系统：

- **0.0-3.9**：低危
- **4.0-6.9**：中危
- **7.0-8.9**：高危
- **9.0-10.0**：严重

## CVE 披露政策

### 1. 披露时间线

- **发现漏洞**：立即开始修复
- **修复完成**：申请 CVE 编号
- **安全公告**：修复后 30 天内发布
- **公开披露**：根据漏洞严重程度确定

### 2. 负责任的披露

我们鼓励负责任的漏洞披露：

- 给维护者合理时间修复
- 不公开漏洞细节直到修复完成
- 配合修复验证

### 3. 豁免情况

以下情况可能延迟披露：

- 需要协调的多项目漏洞
- 涉及第三方组件的漏洞
- 需要额外时间修复的复杂漏洞

## CVE 修复流程

### 1. 修复优先级

- **严重/高危**：24 小时内发布紧急补丁
- **中危**：7 天内发布修复
- **低危**：下一个版本中修复

### 2. 修复验证

所有安全修复都需要：

- 单元测试验证
- 集成测试验证
- 安全测试验证
- 回归测试验证

### 3. 向后兼容性

安全修复优先于向后兼容性，但会尽量：

- 提供迁移指南
- 保持配置兼容
- 提供降级路径
